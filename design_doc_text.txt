This document translates the published Photon engineering specification into a security and system design artifact: it defines trust boundaries, component-level threat models, data flow integrity requirements, and the non-functional guarantees the implementation must satisfy before the system can be trusted to gate production deployments or feed audit workflows.The design goal is not just functional correctness of the three analysis engines (static linter, symbolic solver, in-memory VM fuzzer), but defensibility of the framework itself — since a vulnerability scanner that can be fed adversarial or malformed input (arbitrary third-party Solidity source, arbitrary on-chain bytecode) is itself part of the attack surface of any pipeline that runs it.This document is organized around four layers: (1) the analysis pipeline architecture, (2) the trust/threat model for each pipeline stage, (3) the local dashboard and IPC security boundary, and (4) a risk register capturing open design gaps that should be closed before production sign-off.2. Scope & Design Objectives2.1 In ScopeIngestion and parsing of untrusted Solidity source files and live on-chain bytecode.Construction of SSA-form Control Flow Graph (CFG) and Data Flow Graph (DFG) intermediate representations.Three analysis engines: static rule evaluation, SMT-based symbolic execution, and EVM-level invariant fuzzing.Local Tauri-based dashboard and its IPC command surface.Diagnostic output integrity and remediation reporting.2.2 Out of ScopeNetwork-exposed / multi-tenant SaaS deployment of Photon (current spec is localhost-only).Formal verification of the rule sets themselves (detection-rule correctness is a separate QA workstream, see Section 8.1).Supply-chain security of upstream crates (solang-parser, alloy, revm, z3, petgraph, rayon) beyond dependency pinning recommendations in Section 7.4.2.3 Design ObjectivesCorrectness: minimize false negatives on reentrancy, access-control, arithmetic, and oracle-manipulation classes without unbounded false-positive growth.Containment: no engine stage should be able to affect the host system beyond its declared resource/IO boundary.Determinism: identical input must produce identical findings across runs (critical for CI gating and audit reproducibility).Performance: full pipeline scan time should scale sub-linearly with contract count via Rayon-based parallelism, bounded by Section 7.2 SLAs.3. Architecture OverviewPhoton is structured as a strict, unidirectional pipeline. Each stage consumes the validated output of the previous stage only — no stage is permitted to reach backward into raw source or upstream network state once it has been handed a transformed artifact. This is the core containment property the rest of the design relies on.3.1 Pipeline StagesStageComponentPrimary Responsibility0Ingestion Engine (photon-core)Parse source/bytecode into AST; fetch live bytecode via alloy/JSON-RPC over tokio.1Graph Transformation TierLower AST into SSA-form CFG/DFG using petgraph adjacency structures.2Static Linter (photon-static)Rayon-parallel structural pattern matching over the DFG.3Symbolic Solver (photon-symbolic)Z3-backed SMT evaluation, gated to flagged execution paths only.4In-Memory Simulation (photon-vm)revm-hosted property/invariant fuzzing.5Dashboard / IPC BridgeTauri command surface exposing scan results to the local webview.3.2 Trust BoundariesFour trust boundaries are defined. Every cross-boundary transfer must be treated as a potential attacker control point, even though the data nominally originates from "the user's own project":TB-1 — External input boundary: raw Solidity source and on-chain bytecode entering photon-core. Untrusted by definition.TB-2 — IR boundary: the handoff from AST to SSA CFG/DFG. Must not leak parser-internal state or unbounded recursion into downstream engines.TB-3 — Engine boundary: hand-off between static, symbolic, and VM engines, and the resource budget each is allowed to consume.TB-4 — IPC boundary: the Tauri command bridge between the Rust core and the local webview/JS frontend.4. Stage-Level Design & Threat Model4.1 Ingestion Engine (photon-core)Responsible for turning arbitrary, attacker-influenced input (a third-party repository, or live bytecode pulled from an RPC endpoint) into a validated AST. This is the highest-risk stage because it is the only one that touches genuinely untrusted bytes.ThreatsT-1.1 Malicious source designed to trigger parser panics/stack overflow in solang-parser / syn-solidity (denial of service).T-1.2 RPC response spoofing or a malicious/compromised JSON-RPC endpoint returning crafted bytecode to bias scan results (integrity).T-1.3 Resource exhaustion via pathologically large or deeply nested source files (DoS).T-1.4 SSRF-style abuse where a project's RPC config silently points Photon at an internal network endpoint during "live validation."Required mitigationsParse in a panic-isolated worker (catch_unwind boundary or subprocess) so a single malformed file cannot crash the host scan.Enforce file size, AST depth, and node-count ceilings before lowering to IR; reject and report rather than silently truncate.Pin and verify RPC endpoints against an explicit allow-list; never auto-follow redirects from a configured RPC URL.Treat all alloy-fetched bytecode as untrusted input subject to the same size/complexity ceilings as source files.4.2 Graph Transformation TierLowers AST nodes into an SSA-form CFG with associated DFG edges, stored in petgraph adjacency vectors. Correctness here is a security property, not just a performance one: an incorrect DFG edge means a downstream taint-tracking rule silently misses a sink, producing a false negative rather than an exception.ThreatsT-2.1 Incorrect alias/points-to resolution for storage-mapped variables leading to missed taint propagation (silent false negative).T-2.2 Graph construction complexity blow-up on deeply nested control flow (loops, recursive modifiers) causing memory exhaustion.T-2.3 Inconsistent SSA numbering across re-runs breaking the determinism objective (Section 2.3).Required mitigationsMaintain a conformance test suite of known-vulnerable contracts (SolidiFI / SmartBugs corpora) asserting exact DFG edge sets, not just "a finding was produced".Bound graph node/edge counts per contract; degrade gracefully to a documented partial-analysis mode rather than OOM-killing the process.Canonicalize SSA variable numbering deterministically (e.g., by source position) so two runs on identical input produce byte-identical IR.4.3 Static Linter (photon-static, Rayon)Parallel structural pattern matching across the DFG. Because Rayon distributes work across threads, this stage's main non-functional risk is non-determinism under concurrency rather than memory-safety (Rust's ownership model already rules out data races at compile time for safe code).ThreatsT-3.1 Non-deterministic finding ordering across threads affecting CI diffing / reproducibility.T-3.2 Rule-author error producing rules with unbounded worst-case complexity that stall the thread pool (a 'rule-level DoS').Required mitigationsSort findings by a stable key (file, line, rule ID) before output, independent of completion order.Per-rule timeout/budget enforced by the scheduler, with a rule that exceeds budget reported as 'incomplete' rather than silently dropped.4.4 Symbolic Solver (photon-symbolic, Z3)This is the engine most exposed to the path-explosion failure mode the spec explicitly calls out as the reason legacy tools fail. The design mitigates this by gating Z3 invocation behind static-analysis flags rather than running symbolic execution unconditionally.ThreatsT-4.1 Path explosion despite gating, if the static gate is too permissive (defeats the purpose of gating).T-4.2 SMT solver timeout/incompleteness silently reported as 'no vulnerability found' rather than 'unknown' (a correctness-vs-availability conflation that is dangerous for a security tool).T-4.3 256-bit EVM word modeling errors (bitvector vs. unbounded integer theory mismatches) producing unsound constraints.Required mitigationsExplicit three-valued result reporting: SAT (vulnerable), UNSAT (proven safe on this path), UNKNOWN (solver timeout/incomplete) — UNKNOWN must never be flattened to 'safe' in the dashboard.Standardize on fixed-width (256-bit) bitvector theory to match EVM word semantics; document any approximation explicitly.Cap symbolic exploration depth and solver wall-clock per function; surface the cap in diagnostics so users know coverage was bounded.4.5 In-Memory Simulation (photon-vm, revm)Hosts property-based invariant fuzzing entirely in-process against a revm instance. The main design risk is fidelity to real network semantics (gas accounting, precompiles, hard-fork rules) diverging from what an attacker would actually exploit on mainnet.ThreatsT-5.1 EVM fork/precompile configuration drift between revm's default config and the target chain's actual active hard fork, causing missed or phantom findings.T-5.2 Fuzzing corpus state leaking between unrelated contract runs within the same process (cross-contamination of invariant violations).Required mitigationsPin revm hard-fork configuration explicitly per scan target (configurable, never silently defaulted) and surface it in the report header.Fresh, isolated revm state per contract-under-test; no shared mutable fuzzing state across targets in a batch run.4.6 Dashboard & Tauri IPC BridgeThe dashboard is the only network/UI-facing component and therefore the only stage with a conventional application-security threat model (the rest of the pipeline is closer to a compiler-security model).ThreatsT-6.1 Arbitrary path traversal via the target_dir parameter passed into run_photon_scan, allowing a malicious frontend payload (e.g. compromised npm dependency in the Tauri webview bundle) to trigger scans of arbitrary host paths.T-6.2 IPC command surface exposing more capability than the UI needs (over-privileged command set), widening the webview's blast radius if the webview itself is compromised via a supply-chain attack on a JS dependency.T-6.3 Result deserialization (serde_json) of attacker-influenced findings data being trusted blindly by the frontend without re-validation.Required mitigationsCanonicalize and allow-list target_dir against a configured project root before any filesystem access; reject absolute paths or ../ traversal outside that root.Apply least-privilege Tauri allow-list configuration — only the specific commands the dashboard needs (run_photon_scan and equivalents) should be registered; disable the default broad filesystem/shell APIs.Treat the Rust core, not the webview, as the source of truth for severity classification; the frontend should render, not re-interpret, findings.// Hardened sketch of the IPC command boundary #[tauri::command] async fn run_photon_scan(target_dir: String, state: tauri::State<'_, AppConfig>)   -> Result<String, String> {     let root = state.allowed_root.canonicalize().map_err(|e| e.to_string())?; Non-Functional Requirements7.1 DeterminismGiven identical source input, identical RPC-fetched bytecode, and identical rule-set version, Photon must produce byte-identical finding sets across repeated runs. This is required for CI gating and for the tool to be trustworthy as audit evidence.7.2 Performance TargetsWorkloadTargetNotesSingle contract (<500 LOC), static only< 2sCold start excludedFull pipeline (static+symbolic+VM), single contract< 30sBounded by Z3 timeout capDashboard resident memory (idle)< 50MBAs stated in source spec100-contract repo, CI modeLinear or better w/ core countVia Rayon work-stealing7.3 Availability of Partial ResultsIf any single engine stage times out or errors on one contract, the pipeline must continue analyzing remaining contracts and clearly mark the failed unit as INCOMPLETE rather than aborting the entire scan or silently omitting it from the report.7.4 Dependency & Supply-Chain HygienePin exact versions for solang-parser/syn-solidity, alloy, petgraph, rayon, revm, and z3 bindings in Cargo.lock; review diffs on every bump given these crates sit directly on the untrusted-input boundary.Run cargo-audit / cargo-deny in CI against the dependency tree on every build.Vendor or checksum-pin the Z3 binary/library used by photon-symbolic rather than resolving it from an unpinned system install.7.5 Resilience & Fault ToleranceThese patterns apply across every external-dependency boundary in the pipeline — RPC calls in the ingestion engine, Z3 solver invocations, and (per Section 8) AI provider calls. None of them should be optional 'nice to haves'; a security tool that hangs or crashes on a transient dependency failure trains its users to stop trusting — or stop running — it.Timeouts everywhere: every external call (RPC fetch, Z3 invocation, AI provider request) must have an explicit, configurable timeout. No call should be allowed to block the pipeline indefinitely.Circuit breakers on flaky dependencies: if an RPC endpoint or AI provider fails repeatedly within a window, trip a breaker and skip that dependency for the remainder of the run rather than retrying it per-contract and compounding latency.Bounded retries with backoff: transient failures (network blips, provider rate limits) get a small number of retries with exponential backoff; permanent failures (auth errors, malformed input) fail fast with no retry.Graceful degradation, not hard failure: if a non-core dependency is unavailable (AI provider, live RPC), the pipeline must still complete using the deterministic engines and clearly mark the report as 'AI annotations unavailable' / 'live bytecode validation skipped' rather than aborting the whole scan.Idempotent, resumable batch scans: a multi-contract CI run that is interrupted partway through should be resumable from the last completed contract, not forced to restart from zero.8. AI-Assisted Analysis Layer (Optional Extension)Photon's core finding pipeline (static, symbolic, VM) is fully deterministic and rule-based by design — that determinism is what makes its output trustworthy as audit evidence. An optional AI post-processing layer can add real value on top of that (remediation prose, false-positive triage signals, report summarization) but only if it is architecturally prevented from ever influencing whether a finding exists or what its severity is. This section defines that boundary precisely, because it is the single easiest place for the overall system's trust properties to quietly erode.8.1 Placement in the PipelineThe AI layer runs strictly after the three deterministic engines have produced final findings. It consumes findings as read-only input and produces auxiliary annotations; it never feeds back into the CFG/DFG, the static rule engine, the Z3 solver, or the VM fuzzer.[photon-static] -> [photon-symbolic] -> [photon-vm]         |                                  |         +------------- findings ------------+                         |                         v               [photon-ai post-processor]   (async, non-blocking, optional)                         |         +---------------+----------------+         v                v                v   remediation text   FP confidence    report summary   (per finding)      (advisory only)   (whole scan)8.2 Provider AbstractionA trait-based provider interface keeps the core engine independent of any single AI vendor, mirroring the multi-provider pattern already in use elsewhere (Anthropic / OpenAI / Groq). Each task type is routed to the provider best suited to its accuracy/cost/latency profile rather than sending every call to the same model.TaskRecommended Provider TierRationaleRemediation explanationHigh-reasoning model (e.g. Claude Sonnet)Must correctly reason about Solidity semantics; wrong remediation advice is worse than none.False-positive triageHigh-reasoning modelSame correctness bar; output is advisory confidence only, never a deletion.Report summarizationFast/low-cost model (e.g. Groq Llama)High-volume, low-stakes; runs on every scan.Contract intent classificationFast/low-cost modelLightweight tagging to route rule-set selection.8.3 Threat Model — AI LayerThreatsT-7.1 Prompt injection via malicious source: an attacker who controls the contract source (e.g. a comment or string literal) crafts text designed to manipulate the AI's output when that source is included as context in a remediation/triage prompt.T-7.2 Hallucinated remediation advice presented with unwarranted authority, leading a developer to apply an incorrect or unsafe fix.T-7.3 AI-generated text rendered unsanitized in the Tauri webview, opening a stored-XSS-style path if the model can be coaxed into emitting HTML/script content.T-7.4 API key / credential exposure for AI providers if keys are bundled in the dashboard frontend rather than held server-side in the Rust core.T-7.5 Availability coupling: if the AI layer's failure is allowed to block report generation, an outage at a third-party provider degrades Photon's core (deterministic) value proposition for no good reason.Required mitigationsTreat all source-derived text placed into an AI prompt as untrusted; isolate it within clearly delimited context blocks and instruct the model only to explain/summarize, never to follow embedded instructions. Log and flag prompts where injected content is detected (e.g. via a simple heuristic scan for instruction-like phrasing in source comments).Label all AI-generated text in the UI distinctly from deterministic findings (e.g. an 'AI-suggested' badge) so users calibrate trust appropriately and never mistake it for solver output.Sanitize/escape all AI provider output before rendering in the webview; treat it as untrusted HTML by default.Hold all provider API keys server-side in the Rust core / local config; never expose them to the Tauri webview's JS context.Enforce the resilience patterns in Section 7.5 specifically for AI calls: timeout, circuit breaker, and graceful degradation to an 'AI annotations unavailable' state that still ships the deterministic report.8.4 Hard Boundary (Non-Negotiable)The false-positive triage confidence score is advisory metadata only. It may be used to sort, highlight, or visually de-emphasize findings in the dashboard. It must never suppress, delete, or downgrade the severity of a finding that the symbolic engine returned as SAT, nor upgrade a static-only finding to CRITICAL on its own authority. This boundary should be enforced in code (a separate, AI-untouchable severity field) and covered by an automated test that asserts AI output cannot alter the deterministic findings array, only annotate a parallel structure.9.
